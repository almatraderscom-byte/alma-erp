## Corrections / decisions the roadmap must respect

1. **Head model wiring (context for "Grok 4.20 remains the head").** The env default is still `gemini-3.1-pro` (`head-router.ts:46`); Grok runs because the owner pins `or-grok-4.20` per conversation via `AgentConversation.modelId` (registry entry added 2026-07-12). CLAUDE.md still documents the Gemini-head decision — when the owner confirms Grok as the standing head, CLAUDE.md and/or `HEAVY_HEAD_MODEL_ID` should be updated together; until then treat "Grok head" as per-conversation, not global.
2. **`required`-less schemas = 86, not 73.** Direction identical, Phase 2 scope slightly larger.
3. **Prod vs preview routing differ today.** Phase 3 ("replace first-200 truncation with intent/state-selected packs") must account for BOTH paths: prod fixed-set and the preview dynamic selector, and must not resurrect the 2026-07-08 cost problem the fixed set was built to avoid (that motivation is dead anyway — Grok/Gemini/DeepSeek heads recorded zero Anthropic-style cache hits; Grok caching is automatic prefix-based).
4. **`cache_control` on OpenRouter is harmless but misleading** — OpenRouter forwards it only to Anthropic models; for Grok it's ignored. Removing it for Grok (Phase 3) is cleanup, not a behavior fix.
5. **Registry lists `or-grok-4.20` at 2M context** — matches OpenRouter's page; xAI's own card says 1M. Roadmap's advice (read live metadata, conservative budget) stands.
6. **Owner rules already in force that the roadmap must not regress:** Bangla-only owner output, "Boss" address (no "Sir"), approval cards for writes/spend, `bangla-output-gate.ts` for customer-facing quality, whole-taka money via `roundMoney`.

## Phase 4 status (WorkflowRun unified state — 2026-07-14)

- ✅ **Canonical `WorkflowRun` (`AGENT-STATE-001`)**: `workflow_runs` + append-only `workflow_run_events` (additive migration `20260915120000_workflow_run`), nullable `workflow_run_id` links on `agent_pending_actions` (camelCase — legacy table), `agent_ask_cards`, `agent_open_tasks`, `agent_plans`. - ✅ **Auto-create, never model-dependent**: the turn loop's single staged-card hook gives EVERY pending approval card a run (`waiting_owner`, idempotent on card id, kind derived from the card type via the router's own mapping). Ask cards emitted while exactly one run is active get stamped with its id.
- ✅ **Auto-close / sync**: approve + reject routes sync the run to the card's real status (executed→`done`+proof, approved→`waiting_worker`, rejected→`cancelled`); the many per-type execution branches need NO individual hooks because **every turn start reconciles** the conversation's non-terminal runs against their cards' live status (lazy, fail-open).
- ✅ **Resolution precedes routing**: reconciled runs feed (a) the volatile **workflow snapshot note** — exact status/step/legal-next-tools, with the explicit "হ্যাঁ/continue = resume THIS step, restart forbidden" contract — and (b) a new state-router signal (`workflow:<kind>:<status>`, kind is a pack key by construction).
- ✅ **Structured reply resolution (AGENT-IOS-001, server-side)**: an ask-card answer that matches a card carrying `workflowRunId` binds the owner's reply to the EXACT run in the answer note. (Explicit client-side ids from iOS/web can ride later without server changes.)
- ✅ **Execution guard**: approving a card whose run is already terminal returns 409 `workflow_outdated` with a Bangla reason — a stale card can no longer execute (fail-open on lookup errors, blocks only on positive terminal finding).
- ✅ **Auto-checkpoint**: a deadline-truncated turn with tool activity writes its own `waiting_for_owner` checkpoint (linked to the run, dedup by taskRef) and marks the run `deadline_paused` — never relies on the model calling `save_task_checkpoint`/`track_open_task`.
- ✅ **Tests**: [workflow-run.test.ts](../src/agent/lib/__tests__/workflow-run.test.ts) (11) runs the REAL service against an in-memory prisma fake: transition/version-conflict/terminal-auto-close/idempotent-create/sync matrix/reconcile/guard/snapshot-note.
